Les fichiers CFG

C'est quoi un fichier .cfg ?

Les fichiers CFG servent à mettre des paramètres de scripts dans un seul fichier.

Nous avons trois fichiers .CFG sur le serveur :

  • ox.cfg
  • permissions.cfg
  • server.cfg
Contient les paramètres pour les scripts OX
Contient les rôles d'administration ainsi que leurs permissions
Contient des paramètres serveur ainsi que la liste des scripts à démarrer lors de l'ouverture du serveur

ox.cfg​

PAS TOUCHE !

Ce fichier n'a pas besoin d'être édité. Il contient des configurations de base qui ont déjà été programmées par Potato.

permissions.cfg​

danger

Avant de commencer, vous devez comprendre cette section. Je vous recommande grandement de vous exercer à l'utiliser sur votre serveur local.

Les deux sections

Ce fichier contient deux sections : "RESSOURCES" et "ADMINISTRATION"

  • RESSOURCES
  • ADMINISTRATION
Liste de permissions pour des ressources. Par exemple, dans la section "# Ox_lib", on autorise la ressource à ajouter, retirer et modifier des permissions ace.
Liste de permissions pour les administrateurs.

Les permissions ACE et Principals​

Quand on veut donner une permission ACE à un script, à un groupe ou à un joueur, c'est toujours la même chose.

Principals​

Les principals sont simplement des groupes ou un joueur qui contiennent des permissions.

Bonnes pratiques
  • Il est fortement recommandé de donner des permissions à un groupe et ensuite d'ajouter le joueur dans ce groupe.
  • IL NE FAUT JAMAIS donner de permissions directement au joueur.
Ajouter un joueur dans un groupe
-- Le groupe ou joueur enfant va hériter des permissions du groupe/joueur parent
add_principal <GROUPE OU JOUEUR ENFANT> <GROUPE OU JOUEUR PARENT> 

-- Exemple : Ajouter un joueur à l'aide de son identifiant fivem dans le groupe admin
add_principal identifier.fivem:451281 group.admin -- Le joueur va hérité des permissions ACE du groupe admin
Donner les permissions d'un groupe parent à un groupe enfant
-- Le groupe enfant va hériter des permissions du groupe parent
add_principal <GROUPE OU JOUEUR ENFANT> <GROUPE OU JOUEUR PARENT> 

-- Exemple : Ajouter un joueur à l'aide de son identifiant fivem dans le groupe admin
add_principal group.admin group.mod -- Le groupe admin va obtenir les permissions du groupe mod

ACES​

Les ACES représentent les permissions qu'on donne à un groupe ou joueur.

info

Dans notre cas, les ACES vont seulement nous servir à donner aux groupes des permissions pour effectuer des commandes.

attention

Avec la fonction "addCommand" de Ox_lib, il suffit de spécifier quel groupe peut exécuter cette commande.

  • On à donc pas besoin d'ajouter un ACE pour autoriser le groupe admin à utiliser cette commande
Ajouter une permission à un groupe
lib.addCommand('giveitem', {
    help = 'Gives an item to a player',
    params = {
        {
            name = 'target',
            type = 'playerId',
            help = 'Target player\'s server id',
        },
        {
            name = 'item',
            type = 'string',
            help = 'Name of the item to give',
        },
        {
            name = 'count',
            type = 'number',
            help = 'Amount of the item to give, or blank to give 1',
            optional = true,
        },
        {
            name = 'metatype',
            help = 'Sets the item\'s "metadata.type"',
            optional = true,
        },
    },
    restricted = 'group.admin'
}, function(source, args, raw)
    local item = Items(args.item)
 
    if item then
        Inventory.AddItem(args.target, item.name, args.count or 1, args.metatype)
    end
end)
Ajouter une permission à un groupe
-- le groupe va obtenir une permission
add_ace <GROUPE> <commande> allow

-- Exemple : Le groupe admin va pouvoir utiliser la commande car
add_ace group.admin command.car allow

server.cfg​

Les deux sections

Ce fichier contient deux sections : "PARAMÈTRES" et "SCRIPTS"

  • La section PARAMÈTRES est celle orange.
  • La section SCRIPTS est celle rouge.
  • PARAMÈTRES
  • SCRIPTS
Comme le fichier ox.cfg, cette section contient des paramètres pour plusieurs scripts.
Cette section contient la liste des scripts à activer lors du démarrage du serveur.

PARAMÈTRES​

PAS TOUCHE

On ne touche pas à cette section, merci.

SCRIPTS​

attention

Si on ne spécifie pas qu'on veut activer un script, alors celui-ci ne le sera pas à l'ouverture du serveur.

Démarrer un script
ensure <nom du script> --sensible à la case

ensure ox_lib --Va démarrer le script ox_lib
Arrêter un script
stop <nom du script> --sensible à la case

stop ox_lib --Va démarrer le script ox_lib
Redémarrer un script
restart <nom du script> --sensible à la case

restart ox_lib --Va démarrer le script ox_lib

Organiser les scripts​

Comme vous pouvez le voir, certains dossiers sont entre []. Ils servent de dossier pour stocker des scripts.

Si on veut démarrer une ressource qui est dans un dossier entre [], on n'a pas besoin de spécifier son chemin, car le serveur va regarder dans ces dossiers lors du démarrage du serveur.

Donc, comment on organise nos scripts ?

On crée des dossiers de catégories comme dans la photo et on met les scripts reliés à cette catégorie à l'intérieur.
